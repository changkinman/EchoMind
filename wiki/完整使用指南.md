# EchoMind 完整使用指南

---

## 一、部署与启动

### 方式 A：docker compose 全栈部署

```bash
cd EchoMind
vim .env   # 填写 ANTHROPIC_API_KEY

# 构建并启动所有服务
docker compose up -d --build

# 查看日志（看到小熊饼干图案说明就绪）
docker compose logs -f echomind
```

启动后的服务：Redis(6379) + ChromaDB(8001) + Prometheus(9090) + EchoMind(8000) + Nginx(80)

### 方式 B：docker run 开发模式

```bash
# 启动依赖
docker compose up -d redis chromadb

# 构建镜像
docker compose build --no-cache echomind

# 启动 HTTP 服务器（代码挂载，改了立刻生效）
docker run -it --rm \
  --network echomind_echomind-network \
  -p 8000:8000 \
  -e ANTHROPIC_BASE_URL="https://api.deepseek.com/anthropic" \
  -e ANTHROPIC_API_KEY="your_key" \
  -e ANTHROPIC_MODEL="deepseek-chat" \
  -e REDIS_URL="redis://:echomind123@redis:6379/0" \
  -e CHROMA_HOST="chromadb" \
  -e CHROMA_PORT="8000" \
  -e CHROMA_PERSIST_DIRECTORY="/workspace/data/chroma" \
  -v $(pwd):/workspace \
  -w /workspace \
  echomind
```

### 方式 C：CLI 交互模式

在方式 B 命令末尾加 `python api/main.py --cli`，直接在终端对话。

## 二、对话接口

### 发起对话

```bash
curl -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{"message": "我要退款", "user_id": "user_001", "conv_id": "session_1"}'
```

响应：
```json
{
  "conv_id": "session_1",
  "response": "好的，请提供您的订单号，我帮您处理退款申请。",
  "intent": "billing",
  "agent_type": "billing",
  "escalated": false,
  "latency_ms": 1250.3
}
```

### 多轮对话（传入相同 conv_id）

```bash
curl -X POST http://localhost:8000/chat \
  -H "Content-Type: application/json" \
  -d '{"message": "订单号是 #67890", "user_id": "user_001", "conv_id": "session_1"}'
```

系统会自动从 Redis 读取之前的对话记录，意图识别时带入上下文。

## 四、监控 Agent 在线表现

### 监控如何工作

Monitor 在服务启动时自动运行，每 10 秒执行一次采集循环：

```
采集 Orchestrator.get_stats() 和 ToolManager.get_stats()
    ↓
记录到滑动窗口（60 个数据点）
    ↓
Z-score 异常检测（超过 2.5 倍标准差 → 异常）
    ↓
阈值告警检查：
  - Agent 成功率 < 90% → ERROR
  - 工具成功率 < 95% → WARNING
  - Agent 延迟 > 3000ms → WARNING
  - 工具延迟 > 5000ms → ERROR
    ↓
生成优化建议
    ↓
Orchestrator._best_agent() 下次路由时读取最新 routing_score
    ↓
问题 Agent 自动降权 ← 闭环完成
```

### 查看监控数据

```bash
curl http://localhost:8000/monitor
```

响应：
```json
{
  "agent_stats": {
    "general_0": {"total": 15, "success_rate": 0.933, "avg_ms": 820.5, "routing_score": 0.712},
    "technical_0": {"total": 5, "success_rate": 1.0, "avg_ms": 950.2, "routing_score": 0.778},
    "billing_0": {"total": 3, "success_rate": 1.0, "avg_ms": 780.1, "routing_score": 0.815}
  },
  "tool_stats": {
    "knowledge_search": {"total": 8, "success_rate": 0.875, "avg_latency_ms": 320.5, "consecutive_fails": 0, "circuit_state": "closed"}
  },
  "active_alerts": [],
  "suggestions": [
    {"title": "Agent general_0 成功率偏低", "action": "Orchestrator 已自动降低路由权重...", "priority": 8}
  ]
}
```

### 监控指标含义

| 指标 | 含义 | 影响 |
|------|------|------|
| `success_rate` | Agent/工具的成功率 | 直接影响 routing_score |
| `avg_ms` | 平均响应延迟 | 影响 routing_score 的延迟分 |
| `routing_score` | 成功率×0.7 + 低延迟分×0.3 | Orchestrator 选 Agent 的依据 |
| `consecutive_fails` | 工具连续失败次数 | ≥5 触发熔断器 |
| `circuit_state` | 熔断器状态 | open 时拒绝请求 |

### 演示监控闭环

```bash
# 1. 发几条请求产生数据
for i in $(seq 1 5); do
  curl -s -X POST http://localhost:8000/chat \
    -H "Content-Type: application/json" \
    -d "{\"message\": \"测试消息 $i\", \"user_id\": \"test\"}" > /dev/null
done

# 2. 等待一个采集周期
sleep 10

# 3. 查看监控结果
curl http://localhost:8000/monitor | python -m json.tool
```

## 五、运行评测

```bash
curl -X POST http://localhost:8000/eval/run
```

评测流程：
1. 意图识别评测：8 条标注用例 → Accuracy + Macro-F1
2. 对话质量评测：4 条问题 → 真正调用 Orchestrator → LLM-as-Judge 打分
3. 回归检测：与上次评测对比
4. 生成优化建议

响应：
```json
{
  "pass_rate": 0.83,
  "avg_scores": {"intent_accuracy": 0.875, "relevance": 0.88, "accuracy": 0.82, "completeness": 0.79, "helpfulness": 0.85},
  "regressions": [],
  "recommendations": ["意图识别准确率 < 90%：增加 Few-shot 示例"]
}
```

## 七、三种模式对比

| 对比项 | compose 全栈 | docker run HTTP | docker run CLI |
|--------|-------------|-----------------|----------------|
| 对话方式 | HTTP 接口 | HTTP 接口 | 终端输入 |
| 监控 | `/monitor` | `/monitor` | 不支持 |
| 评测 | `/eval/run` | `/eval/run` | 不支持 |
| 知识库 | `/knowledge/*` | `/knowledge/*` | 不支持 |
| 代码热更新 | 需 rebuild | 挂载目录，立刻生效 | 挂载目录，立刻生效 |
