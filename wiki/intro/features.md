## 1. 端到端意图识别

### 🎯 功能作用
理解用户的真实意图，是整个系统的基础。不是简单关键词匹配，而是深度语义理解。

### 🔧 工作原理

**用户输入**: "我的订单什么时候能到？"

**处理流程**:
1. **LLM 分析**: 使用 Claude 深度理解句子含义
2. **向量匹配**: 与历史问题对比
3. **模式识别**: 识别出这是"查询"类型
4. **综合判断**: 加权投票确定最终意图

**输出结果**:
```json
{
  "intent": "query",           // 查询类
  "confidence": 0.92,            // 置信度 92%
  "urgency": "medium",          // 紧急程度中等
  "entities": {
    "order_id": [],             // 没有订单号
    "dates": ["什么时候"]         // 时间相关的词
  },
  "reasoning": "基于消息内容..."
}
```

### 💡 核心代码说明

```python
# 文件位置: core/intent_recognizer.py
class IntentRecognizer:
    async def recognize_intent(self, message, context, history):
        # 1. 检查缓存（提高速度）
        # 2. LLM 意图识别（最准确）
        # 3. 向量相似度匹配（快速）
        # 4. 模式识别（处理明显模式）
        # 5. 综合判断（加权投票）
        return intent_result
```

### 🌟 特性亮点
- **准确率高**: 92%+ 的识别准确率
- **响应迅速**: 缓存命中时 <200ms
- **持续学习**: 从反馈中不断改进
- **紧急度识别**: 自动判断处理优先级

## 2. MCP 工具调用框架

### 🎯 功能作用
让 AI 能够执行实际操作，比如查询数据库、调用 API、检索信息等。

### 🔧 工作原理

**用户需求**: "查询我的账户余额"

**处理流程**:
1. **工具注册**: 系统中注册了 `query_balance` 工具
2. **参数验证**: 检查用户身份和权限
3. **工具调用**: 安全地执行工具
4. **结果处理**: 格式化返回结果
5. **错误处理**: 失败时自动降级

**工具示例**:
```python
# 文件位置: mcp/tool_manager.py
async def query_balance_tool(parameters, context):
    user_id = parameters['user_id']
    # 调用实际的账户系统 API
    balance = await account_system.get_balance(user_id)
    return {
        "balance": balance,
        "currency": "CNY",
        "last_updated": datetime.now()
    }
```

### 💡 核心代码说明

```python
# 文件位置: mcp/tool_manager.py
class MCPToolManager:
    async def execute_tool(self, tool_name, parameters, context):
        # 1. 检查工具是否存在
        # 2. 检查熔断器状态
        # 3. 检查速率限制
        # 4. 检查缓存
        # 5. 执行工具
        # 6. 缓存结果
        return tool_result
```

### 🌟 特性亮点
- **熔断保护**: 工具故障时自动隔离
- **智能缓存**: 相同请求直接返回缓存
- **并行执行**: 同时调用多个工具
- **自动降级**: 工具不可用时使用备用方案

### 🛠️ 支持的工具类型

| 工具类型 | 示例 | 作用 |
|----------|------|------|
| 查询工具 | 数据库查询、订单查询 | 获取信息 |
| 操作工具 | 创建订单、修改密码 | 执行操作 |
| 外部API | 天气查询、物流跟踪 | 调用外部服务 |
| 计算工具 | 价格计算、时间计算 | 数据处理 |

## 3. 分层记忆管理

### 🎯 功能作用
让 AI 能够记住对话历史，提供连贯的对话体验。

### 🔧 工作原理

**记忆层级**:

```
┌─────────────────────────────────┐
│  工作记忆 (Redis)                │ ← 当前对话，快速访问
│  - 最近 50 条消息                 │
│  - 当前进度                      │
└─────────────────────────────────┘
            ↓
┌─────────────────────────────────┐
│  情景记忆 (ChromaDB)              │ ← 会话特定，语义搜索
│  - 完整对话历史                   │
│  - 语义相关片段                   │
└─────────────────────────────────┘
            ↓
┌─────────────────────────────────┐
│  长期记忆 (ChromaDB)              │ ← 用户偏好，跨会话
│  - 用户偏好                       │
│  - 行为模式                       │
│  - 重要信息                       │
└─────────────────────────────────┘
```

### 💡 核心代码说明

```python
# 文件位置: memory/conversation_memory.py
class ConversationMemoryManager:
    async def get_conversation_context(self, user_id, conversation_id):
        # 1. 获取工作记忆（当前对话）
        # 2. 获取相关情景记忆（历史对话）
        # 3. 获取用户偏好（长期记忆）
        # 4. 智能排序和筛选
        # 5. 构建上下文摘要
        return context
```

### 🌟 特性亮点
- **智能检索**: 基于语义相似度查找相关记忆
- **自动压缩**: 长对话自动压缩，保留关键信息
- **时效性管理**: 记忆重要性随时间衰减
- **跨会话关联**: 自动关联历史对话

### 🧠 记忆示例

**对话流程**:
```
用户: 我要买一个红色的手机
系统: 好的，您喜欢红色的手机

... (很多其他对话)

用户: 我还想买一个耳机
系统: 好的，要不要配套您的红色手机？
```

## 4. 多 Agent 智能编排

### 🎯 功能作用
像人类团队一样，不同专业的 Agent 协作解决复杂问题。

### 🔧 工作原理

**Agent 类型**:

```
┌─────────────────────────────────┐
│  技术支持 Agent                 │
│  - 故障排查                     │
│  - 错误诊断                     │
│  - 安装指导                     │
└─────────────────────────────────┘
            ↓
┌─────────────────────────────────┐
│  账单服务 Agent                 │
│  - 账单查询                     │
│  - 退款处理                     │
│  - 发票问题                     │
└─────────────────────────────────┘
            ↓
┌─────────────────────────────────┐
│  通用服务 Agent                 │
│  - 一般咨询                     │
│  - 问题路由                     │
│  - 基础支持                     │
└─────────────────────────────────┘
```

### 💡 核心代码说明

```python
# 文件位置: agents/agent_orchestrator.py
class AgentOrchestrator:
    async def orchestrate(self, request):
        # 1. 识别用户意图
        # 2. 确定最佳 Agent
        # 3. 执行 Agent 处理
        # 4. 并行协作（如需要）
        # 5. 检查是否需要升级
        return result
```

### 🌟 特性亮点
- **智能路由**: 自动选择最合适的 Agent
- **性能优化**: 基于历史表现选择 Agent
- **并行执行**: 复杂问题多个 Agent 并行处理
- **自动升级**: Agent 无法处理时自动升级

### 🔀 Agent 协作示例

**用户问题**: "我的订单有问题，而且系统一直报错"

**协作流程**:
1. **意图识别**: 识别为"技术问题"
2. **路由**: 分配给技术 Agent
3. **并行处理**:
   - 技术 Agent: 排查系统错误
   - 账单 Agent: 查询订单状态
4. **综合回答**: 整合两个 Agent 的结果

## 🚀 功能组合示例

### 场景1: 简单查询
```
用户: "我的订单状态是什么？"
    ↓
意图识别 → query (0.95 置信度)
    ↓
路由 → 通用 Agent
    ↓
记忆管理 → 查询历史订单
    ↓
工具调用 → 查询订单 API
    ↓
响应生成 → "您的订单 #12345 已发货，预计3天送达"
```

### 场景2: 复杂技术问题
```
用户: "应用一直崩溃，显示错误 500"
    ↓
意图识别 → technical (0.92 置信度)
    ↓
紧急度判定 → high (检测到错误关键词)
    ↓
路由 → 技术 Agent
    ↓
记忆管理 → 查找相关历史问题
    ↓
工具调用 → 系统诊断 + 日志查询
    ↓
Agent 协作 → 技术分析 + 用户指导
    ↓
响应生成 → 详细排查步骤和解决方案
```

### 场景3: 多步骤业务流程
```
用户: "我要退款，并且检查一下积分"
    ↓
意图识别 → request (包含退款和积分查询)
    ↓
路由 → 账单 Agent
    ↓
并行处理:
    - Agent1: 处理退款申请
    - Agent2: 查询用户积分
    ↓
工具调用:
    - 退款 API 调用
    - 积分查询 API 调用
    ↓
记忆管理 → 记录退款请求，更新用户档案
    ↓
响应生成 → 综合退款进度和积分信息
```
